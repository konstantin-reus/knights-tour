## Structure Overview

**Pattern**: Layer-based flat structure. Each architecture component (C-1 through C-4) maps to a single TypeScript file in the `src/` directory. There are no nested feature directories because the project has exactly 4 modules with no shared subdependencies between them — each module is a single file. This pattern was chosen because the architecture defines 4 components with clear boundaries, and the project will contain ~8 source files total. Deeper nesting would add navigation overhead without organizational benefit.

**Principles**:
1. **1:1 architecture mapping**: Each architecture component is a single `.ts` file in `src/`. The file name matches the component name from the architecture document.
2. **Co-located tests**: Each source file's test lives next to it in the same directory with a `.test.ts` suffix. This follows the Vitest default file discovery pattern and keeps related code together.
3. **Flat by default**: No subdirectories within `src/` unless a module grows beyond 3 files. For this project's scope (4 modules, ~8 files), flat is the correct level of organization.
4. **Root-level configuration**: All config files (`package.json`, `tsconfig.json`, `vite.config.ts`, `index.html`) live at the project root, following Vite's `vanilla-ts` template convention.

**Framework Conventions**: The structure follows Vite's `vanilla-ts` template conventions: `index.html` at the root (Vite's entry point), `src/main.ts` as the application entry (imported by `index.html` via `<script type="module">`), `src/style.css` for global styles, and `public/` for static assets. Vitest discovers test files by the `.test.ts` suffix anywhere in the project.

**Architecture Mapping**: C-1 (Algorithm) → `src/algorithm.ts`, C-2 (Board Renderer) → `src/board.ts`, C-3 (Stats Panel) → `src/stats.ts`, C-4 (Animation Controller) → `src/controller.ts`. The entry point `src/main.ts` bootstraps the application by calling `startVisualization()`.

## Module Organization

### Module: algorithm

- **Architecture Component**: C-1 Algorithm Module
- **Responsibility**: Compute the next knight move using Warnsdorf's rule with random tie-breaking.
- **Public API**:
  - `getNextMove(board: BoardState, current: Position): Position | null`
  - `getKnightMoves(board: BoardState, pos: Position): Position[]`
  - `countAccessibility(board: BoardState, pos: Position): number`
  - `createEmptyBoard(): BoardState`
  - `BOARD_SIZE: number` (constant, 8)
- **Internal Structure**: Single file `src/algorithm.ts` containing pure functions and the `KNIGHT_OFFSETS` constant.
- **Dependencies**: `src/types.ts` (imports `Position`, `BoardState`). No other module dependencies.
- **Test Strategy**: `src/algorithm.test.ts` — unit tests running in Vitest `node` environment (no DOM needed). Tests cover: valid moves from corners/edges/center, Warnsdorf selection correctness, tie-breaking behavior, `null` return when stuck, board creation.

### Module: board

- **Architecture Component**: C-2 Board Renderer
- **Responsibility**: Create and update the 8×8 CSS Grid chessboard DOM.
- **Public API**:
  - `createBoard(container: HTMLElement): HTMLElement[][]`
  - `updateCell(cells: HTMLElement[][], pos: Position, state: CellState, moveNumber?: number): void`
  - `resetBoard(cells: HTMLElement[][]): void`
  - `highlightKnight(cells: HTMLElement[][], pos: Position): void`
  - `CellState` type (re-exported from types)
- **Internal Structure**: Single file `src/board.ts`. Creates `<div>` elements, toggles CSS classes, sets `textContent`.
- **Dependencies**: `src/types.ts` (imports `Position`, `CellState`). No other module dependencies.
- **Test Strategy**: `src/board.test.ts` — unit tests running in Vitest `jsdom` environment. Tests cover: board creates 64 cells, cells have alternating classes, `updateCell` sets correct classes and text, `resetBoard` clears all cells.

### Module: stats

- **Architecture Component**: C-3 Stats Panel
- **Responsibility**: Display and update counters and the speed slider.
- **Public API**:
  - `createStatsPanel(container: HTMLElement): StatsPanelElements`
  - `updateStats(elements: StatsPanelElements, stats: StatsState): void`
  - `createSpeedSlider(container: HTMLElement, onChange: (delayMs: number) => void): HTMLInputElement`
  - `showTourResult(elements: StatsPanelElements, complete: boolean, visited: number): void`
- **Internal Structure**: Single file `src/stats.ts`. Creates counter `<span>` elements and the `<input type="range">` slider.
- **Dependencies**: `src/types.ts` (imports `StatsState`, `StatsPanelElements`, `Position`). No other module dependencies.
- **Test Strategy**: `src/stats.test.ts` — unit tests running in Vitest `jsdom` environment. Tests cover: panel creates counter elements, `updateStats` sets correct text values, slider dispatches onChange callback.

### Module: controller

- **Architecture Component**: C-4 Animation Controller
- **Responsibility**: Orchestrate the traversal loop, manage timing, and coordinate algorithm with rendering.
- **Public API**:
  - `startVisualization(config: AnimationConfig): void`
- **Internal Structure**: Single file `src/controller.ts`. Contains `startVisualization()`, `generateStartingPositions()`, `runTraversal()`, `onTraversalComplete()`, and manages the `delayMs` variable.
- **Dependencies**: `src/types.ts` (imports `AnimationConfig`, `Position`, `StatsState`), `src/algorithm.ts` (imports `getNextMove`, `createEmptyBoard`), `src/board.ts` (imports `createBoard`, `updateCell`, `resetBoard`, `highlightKnight`), `src/stats.ts` (imports `createStatsPanel`, `updateStats`, `createSpeedSlider`, `showTourResult`).
- **Test Strategy**: `src/controller.test.ts` — unit tests running in Vitest `jsdom` environment with fake timers (`vi.useFakeTimers()`). Tests cover: starting position generation (64 unique positions), traversal step execution, speed slider integration, completion behavior.

### Module: types

- **Architecture Component**: Shared type definitions across all components
- **Responsibility**: Define the TypeScript types and interfaces used across module boundaries.
- **Public API**:
  - `Position`, `BoardState`, `CellState`, `StatsState`, `StatsPanelElements`, `AnimationConfig`
- **Internal Structure**: Single file `src/types.ts`. Contains only type/interface definitions and constants. No runtime code.
- **Dependencies**: None.
- **Test Strategy**: No tests — types are compile-time only.

### Module: main

- **Architecture Component**: Application entry point (not a separate architecture component)
- **Responsibility**: Bootstrap the application — create DOM containers, import CSS, call `startVisualization()`.
- **Public API**: None — this is the entry point, not a library.
- **Internal Structure**: Single file `src/main.ts`. Creates or selects DOM containers, imports `src/style.css`, calls `controller.startVisualization()`, wraps in `try/catch` for unhandled errors.
- **Dependencies**: `src/controller.ts` (imports `startVisualization`), `src/style.css` (Vite CSS import).
- **Test Strategy**: No direct unit tests — main.ts is a thin bootstrap with no logic. The application is tested through the controller and module tests.

## Naming Conventions

### Directories

- **Rule**: kebab-case for all directories
- **Examples**: `knight-tour/`, `src/`, `public/`
- **Exceptions**: `.ai-orchestrator/` — uses the orchestrator's naming convention (not application code)

### Files

- **Source Files**: `[name].ts` where `name` is a lowercase single word matching the architecture component name
  - Examples: `algorithm.ts`, `board.ts`, `stats.ts`, `controller.ts`, `types.ts`, `main.ts`
- **Test Files**: `[name].test.ts` where `name` matches the source file being tested
  - Examples: `algorithm.test.ts`, `board.test.ts`, `stats.test.ts`, `controller.test.ts`
- **Type Files**: `types.ts` — single shared type file (project has <10 types total)
  - Example: `src/types.ts`
- **Style Files**: `style.css` — single global stylesheet
  - Example: `src/style.css`
- **Configuration Files**: Framework-conventional names at project root
  - Examples: `package.json`, `tsconfig.json`, `vite.config.ts`

### Code Entities

- **Types/Interfaces**: PascalCase, no prefix
  - Examples: `Position`, `BoardState`, `StatsState`, `CellState`, `AnimationConfig`, `StatsPanelElements`
- **Functions**: camelCase, verb-first
  - Examples: `getNextMove`, `createBoard`, `updateCell`, `resetBoard`, `highlightKnight`, `createStatsPanel`, `updateStats`, `startVisualization`
- **Constants**: SCREAMING_SNAKE_CASE
  - Examples: `BOARD_SIZE`, `KNIGHT_OFFSETS`, `DEFAULT_DELAY_MS`, `MIN_DELAY_MS`, `MAX_DELAY_MS`
- **Variables**: camelCase
  - Examples: `boardState`, `currentPosition`, `jumpCount`, `positionsChecked`, `solutionsFound`, `delayMs`
- **CSS Classes**: BEM-inspired with `cell--` prefix for cell states
  - Examples: `cell`, `cell--light`, `cell--dark`, `cell--visited`, `cell--current`, `cell--knight`, `board`, `stats-panel`, `speed-slider`
- **Enums**: Not used in this project (use union types and constants instead)

## Dependency Rules

```
Allowed Dependencies:
  controller.ts  -->  algorithm.ts   (orchestrator calls algorithm for next move)
  controller.ts  -->  board.ts       (orchestrator calls renderer to update DOM)
  controller.ts  -->  stats.ts       (orchestrator calls stats to update counters)
  controller.ts  -->  types.ts       (imports shared types)
  algorithm.ts   -->  types.ts       (imports Position, BoardState)
  board.ts       -->  types.ts       (imports Position, CellState)
  stats.ts       -->  types.ts       (imports StatsState, StatsPanelElements, Position)
  main.ts        -->  controller.ts  (bootstrap calls startVisualization)
  main.ts        -->  style.css      (Vite CSS import)

Forbidden Dependencies:
  algorithm.ts   -/->  board.ts, stats.ts, controller.ts, main.ts
    Reason: Algorithm must be pure — no DOM, no rendering, no orchestration dependencies.
    This is architecture design decision DD-1.

  board.ts       -/->  algorithm.ts, stats.ts, controller.ts, main.ts
    Reason: Board Renderer operates on DOM elements only. It receives data via function
    parameters, not by importing other modules. Prevents tight coupling.

  stats.ts       -/->  algorithm.ts, board.ts, controller.ts, main.ts
    Reason: Stats Panel operates on DOM elements only. Speed slider uses a callback
    (inversion of control) instead of importing the controller.

  types.ts       -/->  (any module)
    Reason: Type definitions are leaf dependencies — they define shapes, not behavior.
    Circular type imports are prevented by keeping all shared types in one file.
```

## Key File Descriptions

### `index.html`

- **Purpose**: Vite's entry point. The HTML shell that loads the application. Defines the DOM containers that the controller populates.
- **Contents**: Minimal HTML with `<div id="app">` containing `<div id="board">` and `<div id="stats">`. A `<script type="module" src="/src/main.ts">` tag loads the application.
- **Dependencies**: None (pure HTML).
- **Depended On By**: Vite dev server and build pipeline use this as the entry point.
- **Architecture Reference**: Provides the `boardContainer` and `statsContainer` DOM elements referenced in IC-4 (`AnimationConfig`).

### `src/main.ts`

- **Purpose**: Application bootstrap. The first TypeScript code that executes. Imports CSS, selects DOM containers, and starts the visualization.
- **Contents**: Imports `./style.css` (Vite CSS injection), imports `startVisualization` from `./controller`, selects `#board` and `#stats` elements, calls `startVisualization({ boardContainer, statsContainer })`, wraps in `try/catch` for unhandled errors.
- **Dependencies**: `src/controller.ts`, `src/style.css`.
- **Depended On By**: `index.html` (loaded via `<script>` tag).
- **Architecture Reference**: Application entry point. Implements the error handling strategy (unhandled errors → DOM error message + `console.error`).

### `src/algorithm.ts`

- **Purpose**: Pure algorithmic logic for the Knight's Tour using Warnsdorf's rule. The most important module for correctness — if this is wrong, the visualization is wrong.
- **Contents**: Exports `getNextMove()`, `getKnightMoves()`, `countAccessibility()`, `createEmptyBoard()`, `BOARD_SIZE`, and the internal `KNIGHT_OFFSETS` constant.
- **Dependencies**: `src/types.ts` (imports `Position`, `BoardState`).
- **Depended On By**: `src/controller.ts` (calls `getNextMove`, `createEmptyBoard`), `src/algorithm.test.ts`.
- **Architecture Reference**: Implements C-1 Algorithm Module and design decision DD-1 (pure functions).

### `src/controller.ts`

- **Purpose**: Orchestrates the entire visualization — manages the animation loop, starting position cycling, and coordinates all other modules.
- **Contents**: Exports `startVisualization()`. Contains internal functions `generateStartingPositions()`, `runTraversal()`, `onTraversalComplete()`, and the mutable `delayMs` variable.
- **Dependencies**: `src/types.ts`, `src/algorithm.ts`, `src/board.ts`, `src/stats.ts`.
- **Depended On By**: `src/main.ts` (calls `startVisualization`), `src/controller.test.ts`.
- **Architecture Reference**: Implements C-4 Animation Controller, design decision DD-2 (setTimeout), and design decision DD-3 (Fisher-Yates shuffle).

### `src/types.ts`

- **Purpose**: Central type definitions shared across all modules. Prevents circular imports by being the single source of truth for cross-module types.
- **Contents**: Exports `Position`, `BoardState`, `CellState`, `StatsState`, `StatsPanelElements`, `AnimationConfig`, and constants `BOARD_SIZE`, `DEFAULT_DELAY_MS`, `MIN_DELAY_MS`, `MAX_DELAY_MS`.
- **Dependencies**: None.
- **Depended On By**: All source modules (`algorithm.ts`, `board.ts`, `stats.ts`, `controller.ts`).
- **Architecture Reference**: Defines the data model entities from the architecture (Position, BoardState, StatsState).

### `src/style.css`

- **Purpose**: All visual styling for the application — board grid layout, cell colors, cell state styles, knight indicator, counters layout, and slider appearance.
- **Contents**: CSS Grid definition for the board (`.board { display: grid; grid-template-columns: repeat(8, 1fr); }`), cell base styles (`.cell`), alternating colors (`.cell--light`, `.cell--dark`), state styles (`.cell--visited`, `.cell--current`, `.cell--knight`), stats panel layout (`.stats-panel`), and slider styling (`.speed-slider`).
- **Dependencies**: None (pure CSS).
- **Depended On By**: `src/main.ts` (Vite CSS import injects styles into the page).
- **Architecture Reference**: Implements design decision DD-4 (CSS classes for cell states). Provides the visual layer for C-2 Board Renderer.

### `vite.config.ts`

- **Purpose**: Configuration for both the Vite build tool and the Vitest test runner.
- **Contents**: `defineConfig` with Vitest test settings: `environment: 'jsdom'`, `globals: true`. No Vite plugins needed.
- **Dependencies**: `vitest/config` (imports `defineConfig`).
- **Depended On By**: Vite dev server, Vite build, and Vitest test runner all read this configuration.
- **Architecture Reference**: Enables the tech-stack decisions (Vite 6.x, Vitest 4.x with jsdom).
